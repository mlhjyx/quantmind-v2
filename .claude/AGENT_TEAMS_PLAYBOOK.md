# QuantMind V2 — Agent Teams 任务手册

> 使用方式：按项目进度选择对应任务，复制发给Claude Code执行。
> 前提：已开启 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
> 节奏：Agent Teams辩论方案 → 你审批决策 → 单Agent执行编码 → Agent Teams审计代码 → 循环
> 注意：Agent Teams消耗token多，每个任务约$5-15。用在刀刃上。

## 一、Phase 0 开始前

### 任务1: Phase 0 实现方案分析

```
组建Agent Team分析QuantMind V2 Phase 0的实现方案。

先阅读CLAUDE.md和docs/目录下所有设计文档。

团队分工：
- 数据工程师：分析数据拉取层的实现方案，评估Tushare接口的坑点和性能瓶颈，规划数据验证策略。重点读 TUSHARE_DATA_SOURCE_CHECKLIST.md
- 回测引擎架构师：分析回测引擎的实现方案，重点关注涨跌停/整手约束/T+1建模/确定性保证，找出设计文档中可能遗漏的边界条件。重点读 DEV_BACKTEST_ENGINE.md
- 因子研究员：分析因子计算管道的实现方案，验证预处理顺序的正确性，评估18个Phase 0因子的数据依赖关系和计算顺序。重点读 DEV_FACTOR_MINING.md
- 质量审查员：站在反方角度，挑战其他三个Agent的方案。找漏洞、找矛盾、找文档里说了但实际做不到的地方

每个Agent独立分析后，互相交流发现。质量审查员负责挑战所有人的方案。

最终产出：
1. Phase 0实现的最优执行顺序（不是文档里写的，是你们讨论后认为最合理的）
2. 文档中已发现的矛盾或遗漏清单
3. 每个模块实现时最可能踩的坑，以及预防方案
4. 你们之间有分歧的点，列出不同观点让我来决策
```

**何时用**：Phase 0编码之前
**预期产出**：实施计划+风险清单，审批后交给单Agent执行

## 二、Phase 0 编码期间

### 任务2: 数据拉取完成后——数据质量全面审计

```
组建Agent Team对QuantMind数据库中已拉取的数据做全面质量审计。

先阅读 docs/TUSHARE_DATA_SOURCE_CHECKLIST.md 了解每个字段的正确单位和预期值。

团队分工：
- 单位验证员：逐表检查每个金额/数量字段的单位是否正确。抽样3只股票（600519贵州茅台、000001平安银行、300750宁德时代）与Tushare数据工具的值逐字段比对
- 完整性验证员：检查每张表的数据完整性——每日股票数量趋势有无骤降、klines和daily_basic日期是否对齐、adj_factor覆盖率、退市股是否真的在symbols表里
- 跨表一致性验证员：检查跨表关联是否正确——klines的amount(千元)和moneyflow的amount(万元)是否真的差10倍、VWAP计算是否单位对齐、financial_indicators的actual_ann_date覆盖率
- 异常猎手：找数据中的异常——价格为0或负数、OHLC逻辑错误(high<low)、非ST股单日涨跌>11%、停牌日有数据但不应该有、新股首日数据异常

互相交叉验证后产出：
1. 每张表的质量评分（通过/警告/失败）
2. 发现的具体数据问题清单（含SQL复现）
3. 建议的修复方案
```

**何时用**：Step 3数据拉取完成后，因子计算之前
**预期产出**：数据质量报告，确认数据可信后再做因子

### 任务3: 因子计算完成后——因子有效性验证

```
组建Agent Team验证QuantMind已计算的因子是否有效。

阅读CLAUDE.md中的因子计算规则和DEV_FACTOR_MINING.md。

团队分工：
- 统计验证员：对每个因子跑单因子分析——IC均值/IR/单调性/分组收益，判断哪些因子在2022-2025还有效，哪些已经衰减
- 预处理审计员：验证预处理顺序是否真的按"去极值→填充→中性化→标准化"执行。方法：对一个因子手动按正确和错误顺序各跑一次，比较IC差异
- 相关性分析员：计算34个因子的相关矩阵，找出高度相关的因子对(>0.7)，建议哪些应该去掉或合并。评估因子组合的有效独立因子数
- 魔鬼辩护人：尝试证明"这些因子组合起来也赚不了钱"——用随机打乱的因子做对照实验，检验等权Top-30是否显著优于随机选股

产出：
1. 每个因子的有效性评级（强/弱/失效）和2022-2025分年度IC
2. 建议保留的因子清单和建议淘汰的因子清单
3. 因子相关矩阵热力图和独立因子数估计
4. 随机对照实验结果（策略是否显著优于随机）
```

**何时用**：因子计算完成后，回测之前
**预期产出**：因子筛选建议，确认因子库质量

### 任务4: 回测引擎完成后——回测可信度审计

```
组建Agent Team审计QuantMind回测引擎的可信度。

阅读CLAUDE.md中的"回测可信度规则"（6条硬规则）和DEV_BACKTEST_ENGINE.md。

团队分工：
- 确定性测试员：同一参数跑两次回测，逐日逐字段比较结果是否bit-identical。如果不一致，定位到具体哪行代码引入了随机性
- 边界条件测试员：构造极端测试用例——全部涨停买不进怎么办、全部跌停卖不出怎么办、资金不够买一手怎么办、某天universe为空怎么办、数据缺失一天怎么办
- 对标测试员：用最简单的策略（如"每月买入沪深300等权Top-10"）手动计算预期收益，和引擎输出比对。如果差异>0.1%，逐日排查原因
- 统计审计员：检查bootstrap置信区间实现是否正确、成本敏感性分析是否覆盖4个档位、年度分解是否准确、12项新增指标是否全部实现

产出：
1. 确定性测试结果（通过/失败+原因）
2. 边界条件测试覆盖率和发现的bug
3. 手动对标偏差分析（逐日diff）
4. 统计指标实现完整性检查
```

**何时用**：回测引擎编码完成后，跑正式回测之前
**预期产出**：回测引擎可信度认证，通过后才能相信回测结果

### 任务5: Phase 0 全部完成后——代码全面审计

```
组建Agent Team对Phase 0已完成的全部代码做全面审计。

先阅读CLAUDE.md了解所有约束规则，然后审查backend/目录下所有代码。

团队分工：
- 安全审计员：检查SQL注入风险、.env泄露风险、API无认证暴露、敏感数据日志泄露、git历史中有无.env误提交

- 数据正确性审计员：检查单位一致性（每个跨表计算都对齐了单位吗）、复权计算是否正确、因子预处理顺序是否真的按CLAUDE.md执行、IC是否用了超额收益
- 规范合规审计员：对照CLAUDE.md的所有规则逐条检查——涨跌停can_trade()实现了吗？整手约束有floor吗？bootstrap CI算了吗？成本敏感性做了吗？退市股在Universe里吗？资金T+1建模了吗？每个COMMENT标注单位了吗？

互相challenge后产出：
1. 按严重程度排序的问题清单（P0必修/P1建议/P2优化）
2. 每个问题的具体文件+行号+修复建议
3. CLAUDE.md规则遵守率评分（N条规则中遵守了几条）
```

**何时用**：Phase 0全部功能完成后，进入Paper Trading之前
**预期产出**：P0问题必须全部修复后才能上Paper Trading

3. 建议的调整方案（如果照目前趋势60天达不到标准）

## 附录：Agent Teams使用原则

1. **辩论用Teams，编码用单Agent** — Teams擅长多视角分析，不擅长协作写同一份代码
2. **每次一个任务** — 不要一次发多个任务给Teams，一次专注一个主题
